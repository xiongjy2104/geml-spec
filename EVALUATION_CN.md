# GEML 联合评估

*[English](EVALUATION.md) | 中文*

> 两位来自标记语言世界两端的评审，对 GEML 做一次设计评审，只问一个问题：**我们会不会
> 推荐用 GEML 取代 Markdown 和 HTML？** 本文评估的是 `main` 分支上的当前状态。

## 评审是谁

- **John MacFarlane** —— CommonMark、Djot、Pandoc 的作者。视角是标记语言设计与规范
  严谨性：文法是否确定？规范是否具有规范性？两个独立实现会不会一致？
- **Thariq Shihipar** —— Anthropic Claude Code 团队工程负责人，《HTML 出人意料的有效性》
  作者。视角是智能体输出：这个格式能不能帮模型给人类交付更好的产物？跟 HTML 的运行时
  比，它表现如何？

## 结论 —— 有条件的"是"

我们现在推荐把 GEML 作为**智能体产出的、有类型、可检查的输出格式（一种 IR），用于结构化
文档**——报告、规范、看板、数据加图表的产物——它的回报在于：构建期引用检查和"表格即事实
来源"能一路活到浏览器渲染出的页面里。它**还不是**随手写散文时 Markdown 的平替，也不是
定制一次性交互件的趁手工具。

之前挡住推荐的两件事现在都解决了。文法已经确定，而且有一套被独立实现复现的一致性测试集
——这是 MacFarlane 的闸门。一个 `.geml` 文件现在能渲染了，既能渲染成自包含 HTML 文件，
也能在浏览器里渲染，并把检查显示出来——这是 Shihipar 的闸门。剩下的有限，文末点名。

## 第一部分 —— GEML 做对的地方

- **单一类型化块原语。** 代码、表格、数学、图表、提示框、元数据，全是 `=== type
  {attrs} … ===`。只需学一种形状、模型只需产出一种形状，畸形输出的分支比 Markdown 那
  一堆特例少得多。
- **构建期引用检查，而且送达读者。** 悬空的 `[[#id]]`、重复 id、未知的 `{{meta}}` 键、
  拼错的图表列名都是硬性诊断——而且活到了产物里：CLI 非零退出，浏览器扩展把它画成页内
  横幅。没有哪个轻量标记格式给这个。
- **确定性，挣到了。** 强调用定界符游程 flanking、列表用明文缩进模型，都写进了规范
  （§5.3、§2.1）；EBNF 去掉了"草稿"。**61 个一致性用例由第二个独立实现复现**——这正是
  "每个输入恰好一个解析"的字面验收测试。
- **运行时，而且有两个。** `geml render` 把文档变成一个自包含、可交互的 HTML 文件（内联
  CSS、可排序/筛选的表格、从绑定表格画出的内联 SVG 图表）。一个 Chrome 扩展在浏览器里
  渲染 `.geml`，file:// 和网页都行。
- **表格与图表是单一事实来源**，绑定在构建期受检，数据是投影进图表的，不是复制。
- **实测，不是断言。** 一套公开的生成流利度基准（`geml-parser/bench/`），让当前模型在
  夹具集上跑零样本对带一页 skill。

## 第二部分 —— MacFarlane：标记设计与规范严谨

> 我之前的反对意见是"无歧义"没挣到：列表和强调欠规范，整门语言唯一的定义就是一个正则
> 解析器。这事做完了。两个最难的问题现在由规范里的*明文算法*解决——强调用定界符游程/
> flanking，列表用缩进模型——而且一个仅凭规范写出、不复用任何参考代码的第二解析器，复现
> 了整套一致性测试集。核心已经把这句话挣到了。`=== #id` 命名闭合和 `[^id]: 文本` 脚注
> 定义也堵上了两个真实的坑。
>
> 所以我转为**推荐把 GEML 作为可检查的结构化文档源**。剩下的是广度，不是健全性：
>
> - **公式 DSL 仍是塞在属性里的字符串小语言**（`compute="FY = Q1 + Q2; …"`）。块体行
>   改写试过、被判更乱，所以近期路子是更好的报错加 skill 引导——但可读性这题先挂着，而
>   bench 显示这正是能力强的模型仍会栽的地方。
> - **没有行内 span、没有定义列表**——而它们对 GEML 瞄准的技术文档是家常便饭。
> - **类型注册表封闭**——不改处理器就没法声明带受检语义的新块类型。
> - **转换有损且单向**（只有 `md → geml`，水平分割线被丢）。一个格式的生死系于它的转换
>   器，这块还薄。
>
> 这些都不再是闸门了，是从"好"走向"广"的路。

## 第三部分 —— Shihipar：智能体输出与运行时

> 我之前说 GEML 产出的是人能打开使用的东西=*零*，并据此说不。这句话不成立了。
>
> CLI 渲染器是真的，不是占位：`showcase.html` 是单个自包含文件——内联 CSS、三张可排序/
> 筛选的表格、五张从 `#fy25` 表直接画出的内联 SVG 图表，唯一的网络调用是 KaTeX/Mermaid，
> 而且*只在文档用到时*才加载。一份纯文本加表格的文档确实零网络。图表读的是表格模型，所以
> 单一事实来源的保证活到了输出里。
>
> **浏览器扩展对我这条轴更关键。** 我的 HTML 论点立在两条腿上：浏览器是人人都有的运行时，
> 产物自渲染。之前 GEML 把两条腿都踢掉。扩展把第一条接了回来：`.geml` 在浏览器里渲染，
> 构建期检查以页内横幅送达读者——区分点一路活到了屏幕上，这是通用文本查看器给不了的。
> "通用"诚实的版本不是"浏览器原生解析 GEML"（它不会），而是"产物像笔记本查看器那样借
> 浏览器渲染，并带着自己的检查"。他们把这个拿下了。
>
> 对照我那七条门槛：**自包含可交互运行时——满足。检查存活进产物——满足。渲染器可用性的
> 诚实——满足**（写明了 KaTeX/Mermaid 用到才走 CDN，graphviz/d2/plantuml 显示为源码）。
> **有效解析率——已测，且公布的数字低于我设的线**：带 skill 时特性使用猛涨（haiku
> 17→83%，sonnet 67→100%），但 parse-clean 只有 67–83%，没到我要的高 90%。**杀手级产物
> 对手写 HTML——部分**（格式演示不错，但没有并排对比）。**token 效率——仍未测。往返
> 编辑——仍没有。**
>
> 所以：**从"不"转为有条件的"是"**——推荐把 GEML 作为智能体的、有类型可检查的输出 IR，
> 用于结构化文档那 80%。保留意见，直说：把那一页 skill 带在上下文里、让解析器留在回路里
> （零样本还不行）；小模型上避开 `compute`/`summary` DSL（活的暗雷）；这不是定制件那
> 20%——一次性小部件还是直接要 HTML；而且现在是开发者模式的 CLI/扩展，不是无摩擦安装。

## 第五部分 —— 通往"无保留推荐"的门槛

1. **生成 parse-clean 进高 90%。** 经验上卡在 **compute/summary 公式 DSL**（我俩都点过，
   bench 实锤）外加弱模型围栏嵌套。最干净的修法仍有争议——块体行改写被否——所以这是个
   待解的*设计*问题，不只是个待办。
2. **往返编辑。** 把渲染后的产物（或 md/html）改回源而不损坏。
3. **token 效率实测**（对手写 HTML），并拿一个杀手级产物并排展示。
4. **表达力广度**（MacFarlane）：行内 span、定义列表、可声明块类型。

## 结语

GEML 从"一个我们欣赏的巧妙格式"，变成了"一个我们会推荐用于智能体产出的结构化产物的格式
——前提是带上 skill、让解析器留在回路里"。我们设的两个闸门，它都交付了：单一确定解析，由
一个独立实现证明；以及一个把产物借浏览器渲染、检查还完好的运行时。剩下的清单有限，而其中
最难的一项，已经被点过名了。

—— *John MacFarlane 与 Thariq Shihipar*
