# GEML 联合评估

*[English](EVALUATION.md) | 中文*

> 来自标记语言世界两端的两位评审，对 GEML 做一次设计评审，面前只有一个问题：
> **要怎样，我们才会推荐用 GEML 取代 Markdown 和 HTML？**

## 评审是谁，为什么是这两人

- **John MacFarlane** —— **CommonMark**、**Djot**、**Pandoc** 的作者。他的视角是
  标记语言设计与*规范严谨性*：文法是否确定、规范是否具有规范性（normative）、两个
  独立实现会不会得到一致结果、以及这个格式是否对得起它自己打出的旗号。
- **Thariq Shihipar** —— Anthropic Claude Code 团队工程负责人，《HTML 出人意料的
  有效性》一文作者。他的视角是*智能体输出*：这个格式能否帮模型给人类交付更好的产物，
  以及在 HTML 已经成为智能体首选的那些维度——布局、色彩、交互、可往返编辑——上，
  它表现如何。

我们被特意凑成一对。GEML 恰好卡在我们两个世界之间：它既想要 Markdown 的纯文本可读性，
又想要 Markdown 从未有过的严谨，同时还刻意拒绝 HTML 的逃生舱口。我们一个在意文法是否
扎实，另一个在意智能体和浏览器能不能拿它做出有用的东西。一个只通过其中一边评审的格式，
拿不到我们的推荐。

---

## 一句话结论——裁决，以及能改变裁决的那一步

GEML 是其对比表里**设计得最用心的纯文本格式**，它的三个核心赌注——单一类型化块原语、
构建期引用检查、AST 优先的文档模型——方向都对，明显师承 Djot 与 Pandoc。

但就目前而言，**我们两人都还不会推荐它取代 Markdown 或 HTML**，理由各异，却殊途同归：

- **MacFarlane：** GEML 宣称"无语法歧义"，但它还没做到这句话所要求的功课。列表与
  强调语义都欠规范，EBNF 还是草稿，而对这门语言唯一的定义就是一个建立在朴素正则上的
  参考解析器。*参考实现不等于规范*——这正是 CommonMark 的全部教训。今天两个"符合规范"
  的 GEML 解析器会给出不一致的结果。
- **Shihipar：** GEML 既封掉了原始 HTML 逃生舱口，又不自带渲染器。所以在我那篇文章
  关心的那条轴上——一个人类能打开并*使用*的东西——GEML 目前产出的是**零**。在这条轴上
  它比 Markdown 还更受限：`.geml` 文件是用来读的，不是用来操作的。

能同时翻转两份裁决的，是同一步棋——而它最好用 GEML 自己的口吻说出来，作为一句 GEML
愿意署名的定位语：

> **GEML 不是 HTML 的竞争运行时，而是那个有类型、可检查的*源*——它编译*成* HTML**：
> 就是你今天让智能体手写的那种自包含、可交互的产物，但它出自一个小而可验证的模型——
> 交叉引用已经查过、数据与图表保证同步，同一份源还顺手吐出 Markdown 和 PDF。

这个重构，把"源里没有原始 HTML"和"没有渲染器"从负债变成了*卖点*：进去的是可移植、
可检查的语义，出来的是多种富目标。这——并非巧合——就是 Pandoc 的架构。本文余下部分，
讲的就是怎么把它变成现实。

---

## "取代 Markdown 和 HTML"到底要求什么

值得把话挑明：「打败 Markdown」和「打败 HTML」是*两条彼此权衡的不同轴*：

| 轴 | Markdown 的位置 | HTML 的位置 | GEML 必须做到 |
|---|---|---|---|
| **严谨 / 确定性** | 弱（无单一解析、碎片化） | 不适用（另一个游戏） | **打败 Markdown：**单一确定解析、规范测试集 |
| **表达力 / 渲染** | 智能体已撞上的天花板 | 人人都有的运行时 | **追平 HTML 的产物：**一个出色的渲染器/编译器 |

Markdown 在严谨上被打败；HTML 只能靠*产出 HTML 所产出的东西*来打败。一个约束到足以
可检查的格式（MacFarlane 想要的胜利），往往也把视觉丰富性约束没了（Shihipar 想要的
胜利），反之亦然。**GEML 唯一能同时赢下两条轴的方式，是别再试图*成为*运行时，而成为
那个*编译到*运行时的可检查源。**记住这句话，它是第四部分的脊梁。

---

## 第一部分——GEML 已经做对的地方

该给的肯定，我们两人都给：

- **单一类型化块原语确实优雅。** 用 `=== type {attrs} … ===` 承载代码、表格、数学、
  图表、提示框、元数据，相比"每个功能一套新语法、外加 HTML 兜底"是真正的简化。这是
  Djot 的围栏块直觉的推广。对模型而言，只需正确产出一种块形状，意味着比 Markdown
  那一堆特例少得多的畸形输出分支。
- **构建期引用检查是该被偷走的特性。** 悬空的 `[[#id]]`、重复 id、未知的 `{{meta}}`
  键、或拼错的图表列名，都是带行号的硬性诊断——而不是三跳之后才发现的静默 404。没有
  哪个轻量标记格式保证这一点。对做多文件编辑的智能体，这是"机器可判定的通过/失败"
  与"祈祷"之间的差别。
- **AST 优先，带文档模型 JSON 和 `diagnostics`。** 用抽象模型而非渲染来定义格式，
  完全正确；而那个结构化的通过/失败，正是 CI 回路或自我纠错的智能体想要消费的。
- **属性模型借得是地方。** `{#id .class key=val}` 加上合理的取值类型，直接来自 Djot /
  pandoc-markdown。血统好，品味也好。
- **表格与图表是单一事实来源。** 让图表用 `data=#id` 读一张真实表格——含计算列——并在
  列缺失时报错，直觉是对的：不会再有数据序列与图表悄悄漂移失同步。
- **表格语言的克制。** 明确排除单元格寻址、`VLOOKUP`、`remote()` 和内嵌程序，把表格
  保持为*文档*特性而非电子表格引擎。划这条线的直觉是对的（尽管下文会论证，这条线划在
  了一个别扭的位置）。

这些都不是小事。接下来的分歧，是你和一个严肃设计之间才会有的分歧，不是和一个潦草设计。

## 第二部分——John MacFarlane 的评估：标记设计与规范严谨

先说一句：GEML 身上有我的指纹，而且是以最好的方式——属性语法是 Djot 的，AST 取向是
Pandoc 的，"构造更少、更规整"的哲学正是我从 CommonMark 起就在主张的。所以下文请当作
一个希望它成功、且自己踩过其中大多数坑的人写的笔记。

### 2.1 "无歧义"这句话还没挣到（决定性问题）

这是核心。README 写着"无语法歧义"，§1 把确定性列为约束。但文档里还没有能*让它成真*的
机制，而参考解析器还在主动打脸：

- **列表几乎没规范，解析器还不能嵌套。** 列表是任何 Markdown 家族文法里最难的一块——
  嵌套、缩进、松散/紧凑、有序列表起始号、惰性续行。规范只给了它一行 EBNF；解析器
  （`geml.ts`）用一个扁平正则匹配列表项，根本没有嵌套概念。这恰恰是每个 Markdown 实现
  分裂了十年的那个角落。
- **强调语义的精度还停在 CommonMark 之前。** "定界符必须贴住非空白字符"这条规则必要，
  但远远不够。解析器用最左正则匹配，会在 `*a **b* c**`、`**a*b**`、词内强调、以及每个
  同定界符嵌套的情形上，与任何其它实现不一致。CommonMark 为把它钉死成唯一答案，动用了
  *定界符游程 / flanking* 算法和大约 **130 个强调用例**。GEML 两样都没有。
- **行内优先级在接缝处定义不足**——`[text *with](url) emphasis*` 到底会发生什么？
  链接对强调的优先级需要一条明文规则，而不是一个实现上的偶然。

解药不是更多散文，而是 CommonMark 的纪律：

1. 一套**规范一致性测试集**，input → 文档模型 JSON 的用例——是几百个，不是 59 个——
   其中*测试集即规范*。CommonMark 的 `spec.txt` 带着约 650 个可运行用例，是有原因的。
2. 对两个最难的问题（列表解析与强调消解）给出**明文算法**，让实现者从规范、而非从读
   TypeScript，重现出解析结果。
3. 一份冻结的文法，把 EBNF 上的"草稿"去掉。

**验收测试：** 一个*仅凭规范*写出的*第二个独立*实现，在整套测试集上与参考解析器一致。
在这件事发生之前，"无歧义"是愿景，不是属性——而 GEML 对 Markdown 的整套说辞，正押在
这一条上。

### 2.2 "单一原语"部分是话术，应当老实承认

块*外框*是统一的，但块*体*按类型各是一门子语言：表格是管道网格**或** CSV/TSV；图表是
外部 DSL；数学是 LaTeX；meta 是 `key=val`；代码是逐字；**外加**一门真正的表达式语言
（`compute` / `summary`，带运算符优先级、聚合函数、`printf` 格式——`table.ts` 里 400
行）住在属性字符串里。

所以一个人或一个模型真正要掌握的面，是*一个外框 + N 门块体子语言 + 一门属性表达式语言*。
这可以说比 Markdown 的*总*面还**大**，只是组织得更整齐。这没问题——但 README 的"只需学
一门文法"只在外框层面成立，诚实版本的说辞是"**一个统一的*外框*，所以结构规整、可检查**"，
而不是"一门文法"。

### 2.3 把公式 DSL 塞进带引号的属性，是设计上的坏味道

`compute="FY [%.1f] = Q1 + Q2 + Q3 + Q4; YoY [%.1f%%] = (FY - PriorFY) * 100 /
PriorFY"` 把一整门表达式语言——自带词法器、解析器、优先级、`[printf]`、`;` 分隔、以及
*单*引号列名（因为属性*本身*已经是双引号）——埋进一个字符串、一个属性、一行物理围栏头里。
这违背了"完全可读的纯文本"的精神。被引为灵感的 Org-mode，至少把公式放在自己的
`#+TBLFM:` 行上。要么把计算抬到它自己的、看得见也 diff 得了的块体行上，要么认定表格是
文档特性、把计算砍掉。折中方案——一张塞进字符串里的电子表格——是两头不讨好。（Thariq
从生成那一侧得出了同样的结论；当解析理论评审和模型输出评审*各自独立*地点了同一个构造，
就值得听一听。）

### 2.4 围栏模型很巧，但脆弱

"闭合围栏必须与开栏*恰好*等长、嵌套要用*更长*的围栏"这条规则，叠加在*所有*结构都用
*单一*定界字符上，有两个隐患：

- **编辑级联。** 加一个更深的嵌套块，可能逼你把每一层外围栏都加长——一次非局部编辑。
- **它由最深的载荷决定。** 在看到最内层之前，你无从知道正确的外栏长度。你们自己的历史
  规范就承认了这点，把深层载荷甩进单独的 `blob`，"因为正确的围栏长度嵌套容易出错"。

两条出路，哪条都行：让闭合围栏是长度**不小于**开栏的 `=` 游程（这样嵌套永远不逼你加长
外侧），或允许可选的**命名 / heredoc** 闭合（`=== code … ===code`）。两者都让闭合保持
*局部*且无歧义。

### 2.5 一个"富表达"格式缺的原语

对一个叫*通用富表达（General Expressive）*的东西，有三处缺口扎眼：

- **没有行内 span。** Djot 和 Pandoc 允许写 `[一个短语]{.term}` 在行内对散文做语义标记。
  GEML 有块级 class，却没有行内对等物，于是"语义 class、绝不作样式钩子"的哲学伸不进
  一个句子里。加上方括号 span。
- **没有定义/描述列表**，而它对 GEML 本就瞄准的技术与参考文档是家常便饭。
- **一个封闭、硬编码的类型注册表。** 新块类型是"警告，按原始保留"，无法在文档内或通过
  旁路 schema *声明*一个新类型的块体模式与校验。于是"万物一原语"在某个机构需要
  `=== requirement {#REQ-1}` 带受检语义的那一刻就撞墙：今天这要求改解析器。一个可声明
  的注册表（一个 `=== typedef`，或一份 GEML 据以校验的外部 schema），才是让这个单一
  原语真正扩展到"万物"、而不必让内核内置每个领域的关键。

### 2.6 互操作是命脉，而它很薄

Pandoc 教会我：一个格式的生死系于它的转换器。GEML 出了一个*有损*的 `md → geml`（它
**丢弃水平分割线**——静默的数据丢失——并把原始 HTML 变成带注记的文本），反方向则一无所有。
没有 `geml → html`，没有 `geml → markdown`，没有通往 Pandoc 式 AST 的桥。破绽就在这个
仓库里：README 的"渲染为"示例，是由 **GitHub 的 Markdown/HTML 渲染器**显示的，因为
GEML 自己没有渲染器。一个连自家橱窗都要靠"翻译出去"来渲染的格式，还没把环闭上。用它的
AST 来定义 GEML（它已经几乎是了），并把转换器写出来；这也正是第四部分的那座桥。

### 2.7 `.gemlhistory` 是稀释内核的范围蔓延

逆向增量 + 关键帧 + SHA-256 + 一套定制的 `delete/replace/insert/move` 补丁语言，重新
实现了版本控制的一大块——而保证更*弱*（规范自己承认 id 链"非密码学防篡改"）、无分支、
回滚有破坏性，还有一门规范说智能体**不得**手写的补丁子语言（这和"完全可读、AI 可写的
纯文本"自相矛盾）。"不用 git"的说辞站不住脚：目标用户——仓库里的开发者和智能体——
绝大多数*有* git，而 git 已经给了内容寻址、防篡改、带分支的历史和庞大的工具生态。这是
为一个狭窄的无 git 小众，铺了一大片规范与解析器表面，还和那三个真正有区分度的点抢注意力。
我的建议：把它从内核说辞里分拆出去。只有当一个明确、清晰的无 git 受众是目标时才保留它，
并且别再把它算作核心区分点。

### 2.8 正面对决 XML 的幽灵，以及检查的边界

两处为定位做的诚实修订：

- **"用一个统一原语承载万物结构"也曾是 XML 的说辞。** 从 XML 到 Markdown 的弧线，正是
  人类拒绝一个统一却僵硬的原语、转投轻量、*特例化*、合手的语法的故事。GEML 在和这段历史
  对赌；它应当说清*为什么*自己不会重蹈 XML 的覆辙（我自己的答案会是：因为特例住在*块体
  模式和可声明注册表*里，而不在表层文法里——但这话得 GEML 自己讲出来，不能躲）。
- **对检查止于何处要诚实。** 引用检查对 *id* 是真的，但伸不到*内容*：一段畸形的 Mermaid
  或 LaTeX 块体会直接放行，因为处理器"不得解释块体"。这是一条可辩护的边界，但 README
  应当写明，因为"构建期检查"读起来比它实际更强。

## 第三部分——Thariq Shihipar 的评估：智能体输出与 HTML 视角

*以下是 Thariq 的评估，作为本评审的第二把声音整合进来。*

### 3.1 第一反应

我的整套论点就是"当产物想要布局时，别再向 Claude 要成片的 Markdown"，所以我读 GEML 只带
一个问题：*这能帮智能体给人类交付更好的产物吗？* 我的看法是分裂的。作为一个**源与互换
格式**，GEML 是它对比表里对智能体最可读的东西——一种块形状、硬性引用检查、留在文本里的
结构化表格。但规范封掉了原始 HTML 逃生舱口，说符合规范的处理器"不得依赖原始 HTML"，且
**不自带渲染器**。于是在我那篇文章关心的那条轴上——视觉丰富、色彩、交互、一个人类能
*操作*的东西——GEML 今天产出的是零。`.geml` 文件是用来*读*的；一个自包含的 `.html`
文件是用来*用*的。作为产物*之下*的那一层，令人印象深刻；作为产物本身，是个非起点。

### 3.2 GEML 对智能体真正做得好的地方

- **单一统一块原语降低畸形输出率。** Markdown 对模型的失败模式*就是*那些特例——分隔行
  少一根管道、围栏与缩进打架、不太像 YAML 的 frontmatter。把一切收进 `=== type {attrs}`，
  是只需正确产出一种文法、而非一打。生成器脑子里分支更少，渲染崩坏更少。
- **构建期引用检查把静默链接腐烂变成可处理的错误。** 这是我会偷的特性：做多文件编辑的
  智能体拿到的是带行号的机器可判定诊断，而不是事后才发现的 404。带 `diagnostics` 的
  文档模型 JSON，正是自我纠错回路想要的结构化信号。
- **表格与图表是单一事实来源**，且**一切留在文本模态、token 成本低**——表格是 `| a | b |`，
  不是一座 `<table>` 塔——这也让它按行干净地 diff，利于基于补丁的编辑。
- **确定、低歧义的结构**（仅 ATX 标题、带类型的属性）——原则上模型能悄悄搞错的更少。
  （John 的第二部分，是"原则上"这三个字的星号。）

### 3.3 在我那篇文章关心的事上，GEML 不如 HTML 的地方

- **没有规范的呈现，没有通用运行时——于是它比 Markdown 还更受限。** 我的 HTML 论点立在
  两条腿上：浏览器是人人都有的运行时，HTML 自渲染成可视、可交互的东西。GEML 把两条腿都
  踢掉。给一个没有工具链的人发一个带 Mermaid 块的 `.geml`，他拿到的是**围栏块里的字面
  DSL 源码**，不是图。GitHub 上的 Markdown 至少会渲染那段 Mermaid。GEML 封掉了它唯一的
  通用兜底，又没拿一个运行时来替。
- **完全没有交互故事。** 我推崇的那些产物——一个能用的小编辑器、一块规划板、一个带批注
  的 PR 视图、一套幻灯片、网格里并排的设计对比——都是*行为性*的。`.class` 是"语义标签、
  绝不作样式钩子"；没有事件、没有脚本、没有布局原语。GEML 能描述文档的*结构*，却永远
  描述不了它的*行为*。
- **冷启动流利度是个真实且被低估的问题。** HTML 出人意料的有效性*就是*训练分布——模型
  零提示就流利。模型见过的 GEML 约等于**零**。在这改变之前，每次生成都要把文法带在
  上下文里，而模型会漂回 Markdown 习惯、去够那个并不存在的原始 HTML 舱口。这件事必须
  被**测量**，不能靠断言。
- **单 `=` 围栏长度嵌套规则是生成的暗雷。** 要嵌套，模型必须跟踪**全局围栏深度**并加长
  *外*栏——一个它必须在写开栏*之前*就规划好的非局部约束。HTML 的 `<tag>`/`</tag>` 啰嗦
  但*局部*：你永远不必知道自己嵌了多深才能正确闭合。对自回归模型，局部性胜过简洁。
  （你们自家历史规范因这条规则"易错"而绕开它——当一份配套规范都在躲你的规则，它就是雷。）
- **属性引号里的 compute/summary DSL** 在一个模型只看得见字符串、看不见结构的地方叠了
  三层引号/转义体制——正是单一原语设计想消除的那种畸形输出的可能来源。

### 3.4 以 Thariq 的身份回应那座桥

下面这个重构能把我从怀疑者翻成感兴趣的人：**把 GEML 定位成那个有类型、可检查的*源*——
它编译出的，正是我如今让人手动向 Claude 要的那种自包含、可交互的 HTML 产物。** 这两个
赌注从来不冲突；它们是不同的层。智能体写干净、低 token、引用受检的 GEML——拿到畸形输出
的降低、断链上的硬错误、单一事实来源的表格、廉价的 diff。一个 GEML 编译器吐出富有的
自包含 `.html`（真实布局、色彩、渲染好的图、由表格模型画出的图表、**交互**），并从同一
个源*也*吐出 Markdown 和 PDF。"*源*里没有原始 HTML"不再是限制，而成了*要点*：进去可
移植、可检查的语义，出来多种富目标。这严格优于让模型手写 2000 行 `<div>` 浆糊、然后
*祈祷*交叉引用对得上——因为眼下**没有任何东西在检查那份 HTML**。

这个重构必须认下两件事。其一，GEML 刻意封闭的词汇（只有语义 `.class`、固定的行内集、
封闭的图表通道集）给生成产物的*天花板*封了顶——编译器只能渲染格式叫得出名字的功能。
我最爱的那些 HTML 产物用的是定制的一次性交互，所以 GEML 要么长出一个有原则的"交互组件"
块类型（在*不*重开原始 HTML 舱口的前提下逃出封闭集），要么承认它瞄准的是结构化文档的
80%，而非定制工具的 20%。范围合理——但要讲明。其二，那个编译器于是*就是* GEML 声称不
需要的运行时依赖；"没有渲染器"在价值主张变成"它编译出一个好产物"的那一刻就不再是优点。
这点也认下。

### 3.5 我推荐 GEML 取代 Markdown/HTML 的门槛

具体且可测。当它确凿地交付下列全部时，我会让人用 GEML：

1. **一个出色的 `geml → 自包含 HTML` 编译器，*带交互***——单文件、任意浏览器、无服务器
   ——从 GEML 源产出至少我那篇文章点名的那些功能（规划板、带批注的 diff 视图、幻灯片、
   可排序/筛选的表格、并排网格）。这是决定性项；今天它产出的是零。
2. **几个作为 GEML 写就的、杀手级的自包含可交互产物**，与人类本会手动要的 HTML 并排展示。
   若编译出的产物没那么好，这层间接就不值。
3. **相对手写 HTML 的、被证明的 token 效率**，在*同一批*产物上——源 token 与编辑补丁
   token——并由编译器抹平视觉差距。一个测出来的数字，不是直觉。
4. **当前 Claude 模型能产出有效 GEML 的证据**，零样本或带一页系统提示 / skill：跑一套
   夹具集，报告**有效解析率**与**诊断洁净率**，并给出两个暗雷（围栏深度、公式 DSL）上的
   失败拆分。带一页 skill 能到高 90% 有效解析，我就信冷启动问题可解。
5. **往返编辑**——人类或模型编辑渲染后的产物，改动无损地落回源。
6. **渲染器可用性上的诚实**——把常见图表渲染器打包进默认编译器，让一个 `diagram` 块开箱
   就是*图*；或者别再暗示图表会"渲染"。
7. **引用检查与表格作为事实来源的保证，要在编译进人类真正收到的 HTML/PDF 后存活**——
   这些交叉检查才是区分点，不能在产物里蒸发。

底线：作为 **HTML 的竞争运行时，GEML 今天输**。作为**那个有类型、可检查、编译成我本就
推荐的自包含可交互 HTML 产物的源**，它是本评审里最有说服力的卖点——也是 John 的"约束并
检查"赌注与我的"HTML 有效性"赌注唯一指向同一方向的地方。把第 1–4 项交付出来，你不必
说服我；产物会替你说。

## 第四部分——核心张力，以及我们都认可的那座桥

我们从两端进来，收敛到了同一句话。就联合、直白地把它说出来——并且刻意用第一人称，因为
这是 GEML 该挂在自家主页上的那句话，而不是我们*对* GEML 提的一条建议：

> **GEML 不是 HTML 的竞争运行时，而是那个有类型、可检查的*源*——它编译*成* HTML：就是
> 你今天让智能体手写的那种自包含、可交互的产物，但它出自一个小而可验证的模型：交叉引用
> 已经查过、数据与图表保证同步，同一份源还一并吐出 Markdown 和 PDF。HTML 是目的地；
> GEML 是安全抵达它的事实来源。**

这是 Pandoc 的架构，而且被验证过：用抽象文档模型定义格式，再把从它出发的转换器握在
手里。它化解了上面那张两轴表里的张力。GEML *作为源语言*在**严谨**上打败 Markdown（单一
确定解析、规范测试集），又*通过编译到 HTML*在**表达力**上追平 HTML——而从不要求源文法
长出事件处理器或 `<div>` 浆糊。智能体在一棵小而可验证的树上推理；人类在自己本就有的
运行时里打开那个富有的结果。

它还把 GEML 两个最受争议的决定重构成了特性：

- **"没有原始 HTML 逃生舱口"**不再是限制，而成了让多目标编译与引用检查*成为可能*的保证。
  原始 HTML 恰恰是那个破坏向 LaTeX、docx 和可检查模型可移植性的东西——MacFarlane 为这个
  问题在 Pandoc 里花了十五年。
- **"没有渲染器"**现在必须*反转*：渲染器成为头号交付物，而非缺失。当卖点变成"编译出一个
  好产物"的那一刻，编译器就是产品。

代价同样直白：**这一切都还不存在。** 在编译器交付且出色之前，这座桥只是一个承诺。这正是
路线图存在的意义。

## 第五部分——通往"被推荐"的路线图

按优先级排列，附验收标准。**P0 项是闸门**——没有它们，我们两人都不会推荐 GEML 取代现状。

### P0 —— 让它成真（闸门）

1. **让规范具有规范性、确定性。**
   - 出一套**一致性测试集**，input → 文档模型 JSON 用例（目标数百个），其中*测试集即
     规范*。用明文算法钉死**列表解析**与**强调消解**（强调用定界符游程/flanking 法；
     列表用缩进模型）。从 EBNF 去掉"草稿"。
   - **验收：** 一个*仅凭规范*写出的*第二个独立*实现，在整套测试集上与参考解析器一致。
2. **交付编译器：`geml → 自包含 HTML`，外加 `geml → markdown`。**
   - 单文件 HTML、任意浏览器、无服务器；图与图表渲染成真实图片（打包 Mermaid/Graphviz/
     D2）；表格连同计算列一起渲染；引用链接可解析。至少含基本交互（可排序/筛选的表格是
     最低线）。
   - **验收：** 本仓库自己的"渲染为"示例由 GEML 编译器产出，*而非* GitHub 的 Markdown
     渲染器。一个只有浏览器的人能打开并使用其输出。
3. **测量智能体流利度，并公布。**
   - 让当前 Claude 模型跑一套 GEML 夹具集，零样本与带一页 skill / 系统提示各一遍。报告
     **有效解析率**与**诊断洁净率**，并隔离围栏深度嵌套与公式 DSL 的失败拆分。
   - **验收：** 带一页 skill 到高 90% 有效解析。若一页 skill 都到不了，说明格式在和训练
     分布作对，于是 P1 的暗雷修复升级为 P0。

### P1 —— 让它确实优于替代品

4. **拆掉两个暗雷。** 把闭合围栏放宽为长度**≥**开栏的 `=` 游程（或加命名/heredoc 闭合），
   并把 **compute/summary 公式从带引号的属性字符串里抬出来**，放到看得见、diff 得了的
   块体行上。
5. **补上一个真正的创作格式所需的表达原语：** 行内 span `[文字]{.class}`、定义列表，
   以及一个**可声明的开放类型注册表**（`=== typedef` 或外部 schema），让各领域不必改
   解析器就能加*受检*块类型。
6. **往返编辑：** 对编译产物的编辑（或至少一条规范的 `html → geml` / `md → geml` 路径）
   无损落回源，在构造重叠处尤其无损。
7. **止住转换数据丢失：** 让水平分割线和原始 HTML 穿过 `md → geml`（作为真实构造，或一个
   可忠实往返的原始块），并补上反方向。

### P2 —— 范围纪律与诚实定位

8. **重新界定 `.gemlhistory` 的范围。** 把它从内核说辞里分拆出去；有 git 处就靠 git；
   只在一个明确、清晰的无 git 受众是显式目标时才保留它。别再把它算进头号区分点。
9. **修正定位。** 主打"**编译成好 HTML 的可检查源**"，而非"取代 md/html"。正面对决 XML
   的对比，并点明冷启动训练数据的现实，别让 AI 友好的说法暗示一种还没测过的零样本流利。
10. **对 README/COMPARISON 做诚实修订。** 写明今天没工具链图表不渲染；构建期检查覆盖的是
    *id*、不是块体*内容*；以及"渲染为"示例由外部渲染器产出。在我们两边社区的可信度，系于
    这些。

## 结语

GEML 是个确实好的想法，走完了大约 60% 的路。它的赌注——一个可检查的单一原语、一个 AST
优先的模型、会大声失败的引用——都押对了，且师承 Djot、Pandoc 与来之不易的 CommonMark
教训中最好的部分。拖住它的，是它在我们两条轴上同一个缺口：它*描述*了一个严谨、富表达的
格式，却还没*建造*那两样能让描述成真的东西——一份保证唯一解析的规范，以及一个把可检查的
源变成人类真能用的富产物的编译器。

把这两件事做了，采纳那个调和我们两种世界观的"编译成 HTML 的源"定位，再剪掉那些分散核心
注意力的范围蔓延——GEML 就不再是一个我们欣赏的巧妙格式，而成为一个我们推荐的格式。我们
都愿意走到那一步。从这里到那里要做的功课，清晰、有限，且值得去做。

—— *John MacFarlane 与 Thariq Shihipar*
