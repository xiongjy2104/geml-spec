# 改一行字,不该让 AI 先读完整份文档

*块可寻址的纯文本文档格式:agent 按 `#id` 只读/只改某一节——不 grep、不重读整篇——文档还自带版本历史,不绑 git。*

我最后是手工做完的。

一份越写越大的技术方案——数据仓库迁移与重构的规划,满是表格和图表,而且内容还在不停累加:每让 agent 按一条指令跑一轮,产出的东西就得写回这同一份文档——我拿 Claude Code 一版一版改了好几天,可要交出一个我敢信的版本,唯一的办法还是自己坐下来,把最后一公里手动补完。不是模型不够聪明,是这份*文档*在跟我们俩较劲。

每个小改动,代价都远超它本身。让 agent 调某一节,它得先 grep、把文件一大段拉进上下文找感觉,为定位几行字烧掉成千上万 token。然后它「好心」地做得比我要的更多——重排了我没提过的章节,把两页开外的一张表也重排了。表格和正文渐渐对不上;图里的一个数,和它来源那张表也对不上了。这些没有任何东西拦得住,因为在一份纯文本文档里根本*拦不住*:没有一个单位能让 agent 说「只改这一块」,没有一条纽带把图表绑到它的源表,也没有任何记录告诉你「上一个好版本」长什么样。

到最后,我不是在编辑一份文档,我是在给它当保姆。

## 到底缺了什么

那几个坑其实是同一个缺失换了几张脸:一份纯文本文档没有*可寻址、可校验、带版本的结构*。所以我给它造了一个。GEML 依然是纯文本——你像读 Markdown 一样读它、diff 它——但每一个块都是带类型、可按 `#id` 寻址的单位。就着那份方案坑我的每一处,看看它改变了什么。

**agent 不必再为改一个角落而读完整篇。** 每块一个 `#id`。`geml get #id` 只吐那一块,`geml set #id` 只换那一块、别处一字节不动。拿 GEML 规范本身(它就是用 GEML 写的)来说,按 id 取一块只占全文一个零头:

```console
$ wc -c GEML-spec.geml
19775 GEML-spec.geml
$ geml get GEML-spec.geml '#abstract' | wc -c
633
```

633 字符,而不是 19,775——约 3%,进模型上下文的量少了约 31 倍,而且文档越大、差距越大。因为 agent 压根没加载两页外那张表,它也就无从「好心」重排它。

**数字不再漂移。** 表格自己算列,图表按 id 绑到这张表,而不是把数字复制一份出去:

```
=== table {#fy25 format=csv header=1 compute="FY [%.1f] = Q1 + Q2 + Q3 + Q4"}
Segment,  Q1, Q2, Q3, Q4
Cloud,     8, 10, 12, 14
Platform,  5,  6,  7,  9
===

=== diagram {#rev format=geml-chart data=#fy25 type=bar x=Segment y=FY}
===
```

数值只存一份。加一行,合计和图表都跟着变——没有第二份拷贝可漂,「图和表对不上」从根上不会发生。

**引用要么解析,要么让构建失败。** 用 id 指向任意块——`[[#migration-plan]]`、图表的 `data=#fy25`、一个脚注——`geml check` 会把解析不到的引用变成一个退出码非零的构建错误,而不是一条几周后让读者替你发现的静默死链。

**「上一个好版本」是自带的。** `geml history` 把每个版本存进文件旁边一个纯文本 `.gemlhistory`——提交、原样读出旧版、回滚到任意一版——离线、不绑 git、不依赖服务。就是我一路上一直盼着这份文档能有的版本历史,如今就躺在它边上。

这算不上四个功能,是同一个根:文档需要可寻址、可校验、带版本的结构。其余都从这儿长出来。

## 那为什么不直接用 Markdown?

这才是该问的问题。今天,Markdown 仍是默认——对「人手写、给人读」的散文,它也是个称职的默认:轻、通用、到处能渲染。

问题在于:让 Markdown 到处都能渲染的那份极简,也意味着它不带 agent 需要的东西——可寻址的块、被校验的引用、能算能绑的数据。对工具来说,一份 `.md` 基本就是一根长字符串,没有比「整篇」更小、能稳稳抓住的单位。这些你当然能加扩展补上,但每加一条非标准语法,`.md` 就少一分「到处能渲染」——而那恰恰是 Markdown 唯一真正的王牌。(「那我扩展 Markdown 不就行了」这一问,下面「它不是什么」正面接。)

(有人于是伸手去够 HTML——Anthropic 那篇《the unreasonable effectiveness of HTML》聊的就是这个方向;但 HTML 很重,每个 `<tag>`、每条内联样式都是 token,它是渲染交付物,不是你要的源。岔路,先按下。)

GEML 想做的,是保住 Markdown 好的那部分(纯文本、易读、diff 友好),再补上它没有的那层结构——可寻址、可校验、带版本。而关键在于:它**不是又一种 Markdown 方言,而是一份从头设计的统一规范**——一套语法、一份规范、一套第二实现能复现的一致性测试,同一份文档,谁来解析都是同一个意思。等你要交付,`geml export` 把它投影成 GitHub 风味 Markdown 或 HTML。读写像 Markdown 一样轻,操作起来像一个数据结构一样有把手,最后编译成你读者本就打得开的东西。

## 它不是什么

把话挑明,毕竟你就该对一个新格式带着怀疑:

- **「凭啥又来一个标准?扩展 Markdown 不行吗?」** 你可以试——Pandoc、kramdown 确实加了 `{#id}` 之类。但这正是病根:每次扩展都长出**又一种方言**,同一份 `.md` 在 CommonMark、GFM、Pandoc 眼里含义都不一样;Markdown 本就是一堆互不兼容的方言,再打补丁只会更歧义、更乱。而且一旦用上非标准语法,`.md` 就不再在 GitHub / Notion / Slack 里正确渲染——你为这些能力交出了 Markdown 唯一的王牌。何况引用校验、按 id 取/改块、图表绑表根本不是语法糖,得靠一个带文档模型的解析器加一个构建步骤。根治的办法不是再加补丁,而是从一开始就用一份统一规范约束住它:一套语法、一份 spec、一套第二实现能复现的一致性测试——「什么是对的」由规范说了算,而不是「哪个 parser 碰巧怎么解析」。GEML 就是这么设计的;要到处渲染,`export` 成 Markdown 交付即可。
- **「我会被绑死吗?」** 不会——这正是 `geml export` 的意义。GEML 是你的源,外界照样拿到 `.md` 和 `.html`。先在一份文档上试,随时可退。
- **「模型不会写它。」** 今天有一半是真的——它新,训练数据也薄。两点能缓:语法小而规整(万物同一种类型块),恰是模型能稳定产出的那类东西;而且你随时能像改 Markdown 一样亲手读改,不必指望模型一次到位。
- **它还早,而且刻意做小。** 1.0 规范已稳定,仓库自举。有公开提案流程(GEP),也有一套第二实现能复现的一致性测试。没有采用数据可吹,我也不编。如果 token 成本、漂移的数字、断掉的引用、丢失的版本在大型 AI 改动的文档上咬过你,这就是给你的;要是没有,Markdown 挺好——真的。

## 来试试,顺便验我的数

别信我那个 ~31×:自己 clone 下来,在规范上跑 `geml get #id`——规范就是用 GEML 写的。

- **Playground**,浏览器里、免安装:https://geml-spec.github.io/geml-spec/playground/ ——弄断一个引用,看构建变红。
- **CLI:** `npm i -g @geml/geml`,然后拿你自己最难搞的那份文档跑 `geml get`、`geml check`、`geml history`。
- **仓库和规范:** https://github.com/geml-spec/geml-spec ——欢迎 issue、拍砖,也欢迎用第三种语言写个 parser。

我做这东西,是因为一份我没法交给 agent 的文档,把我逼到了手工收场。我想听听它在你那儿会栽在哪——以及 GEML 哪里撑不住。
